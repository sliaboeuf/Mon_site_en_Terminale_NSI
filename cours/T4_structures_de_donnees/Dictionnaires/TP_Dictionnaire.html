    TP : Implémentation d'un dictionnaire

    Objectif : Dans ce TP, nous allons dans un premier temps écrire plusieurs implémentations d'un dictionnaire. Ensuite, nous comparerons expérimentalement les complexités de différentes implémentations dont celles qui ont été écrites en mesurant leur temps d'exécution. Pour ces implémentations, la taille maximale du dictionnaire est choisie à l’avance, lors de sa création. Par exemple : Dico1(4) générera un objet dictionnaire pouvant contenir \(~2^4=16~\) entrées au maximum. Voici la liste des méthodes nécessaires au bon fonctionnement des dictionnaires quelle que soit leur implémentation :

    DEBUT

    Description

    Les entrée du dictionnaire sont stockées dans une table de hachage qui est une liste de listes :
    • La position de l’élément dans la table de niveau 0 est déterminée à partir de la fonction de hachage de Python.
    • Les collisions sont gérées au moyen d'une liste de niveau 1 qui contient tous les éléments donnant le même résultat par la fonction de hachage.

    Par exemple, pour un dictionnaire de taille 4 contenant 3 éléments :

    🐍 Script Python
    table = [[('trois', 3)], [('un', 1)], [('deux', 2], [ ]]

    La position des éléments se justifie par :

    • hash('un') renvoie -5160269262281110283 et -5160269262281110283 % 4 renvoie 1 donc la clé 'un' sera stockée en position 1.
    • hash('deux') % 4 renvoie 2.
    • hash('trois') % 4 renvoie 0

    Etude des complexités temporelles

    Complexités théoriques

    Dans ce paragraphe, nous allons étudier sommairement les complexités théoriques de 5 implémentations différentes d'un dictionnaire, à savoir :

    • Les 3 dictionnaires que nous venons d'implémenter (Dico1 avec indexation à l'aide de deux listes, Dico2 avec indexation à l'aide d'une liste de tuples et Dico3 avec indexation par fonction de hachage et gestion des collisions par chaînage linéaire).
    • Dico4 dont l'indexation se fait par fonction de hachage et la gestion des collisions par sondage linéaire :
    Code
    🐍 Script Python
    """ Implémentation n°4 d'un dictionnaire en programmation orientée objet Python
    Ce fichier est le squelette de départ d'un code d'une classe Dico4, destinée 
    à produire des dictionnaires.
    Pour cette quatrième implémentation, l'idée est de stocker nos données en 
    déterminant leur position à partir d'une fonction de hachage :
        - Utilisation de la fonction hash de python.  
        - Gestion des collisions par sondage linéaire.
"""

class Dico4:
    """ Dictionnaire contenant une table de hachage """

    def __init__(self, nb_de_bits):
        """ Constructeur : va créer une table dont les (2**nb_de_bits) éléments 
            sont initialisés à None."""
        
        self.longueur = 2 ** nb_de_bits      # nombre total d'éléments
        self.table = [None] * self.longueur
        self.collision = 0                 # compteur de collision

    def _hachage(self, x):
        return hash(x) % self.longueur

    def est_vide(self):
        """ Renvoie True si et seulement si l'objet de type Dico3 est vide, 
            sinon renvoie False"""
        
        for e in self.table:
            if e:
                return False
        return True

    def affichage(self):
        """ Affiche dans la console le contenu du dictionnaire sous la forme 
            suivante :
                    DEBUT
                    indice1 - cle1 : valeur1
                    indice2 - cle2 : valeur2
                    (...)
                    FIN
            Les cellules vides ne s'affichent pas.
            Cette méthode ne renvoie rien."""
        
        ch = 'DEBUT\n'
        for index, valeurs in self.table.items():
            if valeurs:
                if valeurs == 'del':
                    ch += f'{index} - del\n'
                else:
                    cle, valeur = valeurs
                    ch += f'{index} - {cle} : {valeur}\n'
        ch += 'FIN'
        print(ch)
    
    def ecrire(self, cle, valeur):
        """ Ajoute le tuple (cle, valeur) à la position index = hash(cle) dans 
            la table de hachage."""
        
        index = self._hachage(cle)
        if self.table[index] and self.table[index][0] == cle:
            self.table[index] = (cle, valeur)
        else:
            cpt = 0
            while cpt < self.longueur and self.table[index] not in (None,'del'):
                cpt += 1
                index = (index + 1) % self.longueur # pour recommencer en début de liste une fois arrivé à la fin
            if cpt == self.longueur:
                print('Débordement de la table de hachage')
            else:
                self.table[index] = (cle,valeur)
                self.collision += cpt 
        
    def taille(self):
        """ Renvoie le nombre de clés non vides stockées dans le dictionnaire."""
        
        cpt = 0
        for e in self.table:
            if e:
                cpt += 1
        return cpt
        
    def est_plein(self):
        """ Renvoie True si le dictionnaire est plein , sinon renvoie False."""
        
        for e in self.table:
            if not e:
                return False
        return True
        
    def supprimer(self, cle):
        """ Supprime le tuple (cle, valeur) : la case vidée de son contenu est
            alors marquée à 'del' afin de la distinguer d'une case jamais remplie et
            ainsi ne pas corrompre la méthode de gestion des collisions.
            Ne renvoie rien."""
        
        index = self._hachage(cle)
        cpt = 0
        while cpt < self.longueur and self.table[index]:
            if cle == self.table[index][0]:
                self.table[index] = 'del'
            cpt += 1
            index = (index + 1) % self.longueur
        
    def contient(self, cle):
        """ Renvoie True si la clé existe, sinon renvoie False."""

        index = self._hachage(cle)
        cpt = 0
        while cpt < self.longueur and self.table[index]:
            if cle == self.table[index][0]:
                return True
            cpt += 1
            index = (index + 1) % self.longueur
        return False
        
    def lire(self, cle):
        """ Renvoie la valeur correspondante à la clé passée en paramètre.
            Renvoie None si cette clé n'existe pas."""
        
        index = self._hachage(cle)
        cpt = 0
        while cpt < self.longueur and self.table[index]:
            if cle == self.table[index][0]:
                return self.table[index][1]
            cpt += 1
            index = (index + 1) % self.longueur
        return None

    def ecrire(self, cle, valeur):
        """ Ajoute cle en queue de la liste self.cles,
            ajoute valeur en queue de la liste self.valeurs.
            Si la clé existe déjà, seule la (nouvelle) valeur est enregistrée
            à la place de l'ancienne.
            Ne renvoie rien."""

        self.dico[cle] = valeur

    def taille(self):
        """"Renvoie le nombre de cles non vides stockées dans le dictionnaire"""

        return len(self.dico)

    def supprimer(self, cle):
        """ Supprime la valeur correspondante à cle dans la liste self.valeurs,
            Supprime cette clé de la liste self.cles.
            Ne renvoie rien."""

        del self.dico[cle]

    def contient(self, cle):
        """ Renvoie True si la clé existe, sinon renvoie False."""

        if cle in self.dico.keys():
            return True
        return False

    def lire(self, cle):
        """ Renvoie la valeur correspondante à la clé passée en paramètre.
            Renvoie None si cette clé n'existe pas."""

        return self.dico[cle]

    Pour chacune des cinq implémentations, nous allons estimer la complexité temporelle des trois méthodes de classe ecrire, lire et supprimer à partir de l’analyse algorithmique de leur code source :

    • Dico1 :
      Les deux méthodes lire et supprimer font appel à la fonction trouver_index qui implémente une boucle bornée. ⇒ complexité en \(~\mathcal{O}(n)\).
      Concernant la méthode ecrire, en plus de l’appel à la fonction trouver_index il faut également procéder à la recherche du premier emplacement vide (recherche de type séquentielle de complexité linéaire). ⇒ Complexité temporelle globale en \(~\mathcal{O}(n)\).
    • Dico2 :
      La méthode lire fait appel à la fonction trouver_index dont le principe algorithmique est celui de la recherche par dichotomie dans une liste triée ⇒ complexité temporelle en \(~\mathcal{O}(\operatorname{log_2}\)\((n\)\())\). Les méthodes ecrire et supprimer font également appel à la fonction trouver_index, mais elles implémentent en plus une fonction de tri (sort). ⇒ Complexité temporelle globale en \(~\mathcal{O}(n\operatorname{log_2}(n))\).
    • Dico3 :
      Les trois méthodes lire, ecrire et supprimer font appel à la fonction _hachage (coût constant) pour déterminer l’indice correspondant à la position réelle d’une donnée. Compte tenu de la méthode de gestion des collisions (par chaînage linéaire), la recherche d’un élément à une position donnée s’opère de façon séquentielle c’est-à-dire avec une complexité temporelle linéaire. Toutefois, dans l’hypothèse d’un nombre faible de collisions voire d’une absence de collision (cas le plus favorable), l’accès aux données se fait directement à partir de la position calculée par la fonction de hachage (coût constant) puisqu’à chaque élément de la liste correspond alors un unique tuple de données. ⇒ Complexité temporelle en \(~\mathcal{O}(1)~\) dans le cas le plus favorable pour les trois méthodes.
    • Dico4 :
      Les trois méthodes lire, ecrire et supprimer font appel à la fonction _hachage (coût constant) pour calculer l’indice correspondant à la position d’une donnée. Compte tenu de la méthode de gestion des collisions (par sondage linéaire), la recherche d’un élément à partir de sa position calculée s’opère de façon séquentielle c’est à dire avec une complexité temporelle linéaire. Toutefois, dans l’hypothèse d’un nombre faible de collisions voire d’une absence de collision (cas le plus favorable), l’accès aux données se fait directement à partir de la position calculée par la fonction de hachage (coût constant) qui correspond alors à la position réelle dans la liste. ⇒ Complexité temporelle en \(~\mathcal{O}(1)~\) dans le cas le plus favorable pour les trois méthodes.
    • DicoPython :
      Cette implémentation utilise directement le type natif dict de Python (cf documentation ci-dessous) :
    • Types natifs -- Les types de correspondances — dict
      https://docs.python.org/fr/3/library/stdtypes.html#mapping-types-dict
    • PEP 456 -- Secure and interchangeable hash algorithm : SipHash
      https://www.python.org/dev/peps/pep-0456/#siphash
